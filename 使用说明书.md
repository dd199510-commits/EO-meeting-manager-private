# CEO Office 会议管理系统使用说明书

## 1. 产品简介

`CEO Office 会议管理系统` 是一个基于 `React + Vite + Electron` 的桌面应用，用于管理高频会议资料、生成排程清单、接入 AI 生成建议方案、进行人工审核，并输出预留通知文案。

它适合以下场景：

- CEO Office / 总办 / 秘书处维护固定会议台账
- 需要按周、月、年统一管理会议节奏
- 需要先由 AI 给出建议，再由人工确认最终排程
- 需要批量生成“请预留时间”的通知文案

---

## 2. 系统主流程

系统推荐按以下顺序使用：

1. 在“会议库”中维护会议主数据
2. 生成指定时间范围内的待排程会议清单
3. 配置 AI 排程偏好并生成建议排程
4. 将 AI 结果导入“审核调整”进行人工确认
5. 在“最终检查”中核对是否遗漏或多排
6. 在“预留通知”中复制通知文案并标记发送
7. 通过“记录”查看操作痕迹，必要时使用系统备份导出/恢复

---

## 3. 界面模块说明

### 3.1 顶部导航

系统顶部提供三个一级模块：

- `会议库`：维护会议主数据
- `排程`：完成生成清单、审核调整、最终检查、预留通知
- `记录`：查看操作审计日志

右上角的 `系统备份` 支持：

- `导出系统备份`
- `恢复系统备份`

---

## 4. 会议库使用说明

### 4.1 会议列表

会议库用于维护所有会议的基础资料。支持：

- 搜索会议名称或参会人
- 按频率、时间范围、历史状态等条件筛选
- 按频率、下次会议、最近历史、名称、自定义顺序排序
- 在“自定义排序”下拖拽调整会议顺序
- 查看不同频率会议数量汇总

### 4.2 新建 / 编辑会议

点击 `新建会议` 或进入已有会议的编辑页，可维护以下内容：

- 会议名称
- 会议频率：`周会 / 月会 / 年会 / 不定期`
- 周期参数
  - 周会：每几周、星期几
  - 月会：每几个月、每月几号
  - 年会：每几年、哪些月份、每月几号
- 会议时长
- 锚点日期
- 参会人
- 会议备注
- 备注关联对象
- 通知模板配置
- 历史召开记录

说明：

- `锚点日期` 是周期推算的起点，用于计算未来发生日期
- 当设置 29、30、31 号时，如当月天数不足，系统会自动调整到当月最后一天
- `不定期` 会议不参与自动周期推算

### 4.3 批量导入历史记录

点击 `批量导入历史` 后，可直接粘贴从表格复制出来的三列制表符数据：

- 会议主题
- 开始日期
- 开始时间

系统会先自动匹配已有会议：

- 匹配成功的记录进入“可导入”
- 匹配失败的记录可手动指定对应会议
- 如果日期已存在于历史记录中，系统会提示重复

这个功能适合补录过去一段时间的会议历史，帮助系统更准确理解会议节奏。

### 4.4 删除与回收站

删除会议后，会议不会立即彻底消失，而是进入 `回收站`。在回收站中可以：

- 恢复会议
- 彻底删除

建议先恢复确认，无需直接做永久删除。

## 5. 排程模块使用说明

排程模块分为四个子步骤：

- `生成清单`
- `审核调整`
- `最终检查`
- `预留通知`

### 5.1 生成清单

这一页的目标，是先生成某个时间范围内需要安排的会议实例。

主要操作：

- 选择排程时间范围
- 生成当前范围内的会议实例
- 查看日历视图或实例明细
- 导出待排程数据

系统会把每个实例转成一条待排程任务，后续 AI 和人工审核都基于这份批次快照进行。

如果会议库内容发生变化，而你之前已经生成过清单，系统会提示当前清单已过期，建议重新生成。

### 5.2 配置 AI 与排程规则

系统支持两类 AI 供应商：

- `OpenAI`
- `Gemini`

可配置内容包括：

- 选择供应商
- 选择模型
- 设置 API Key
- 添加排程规则
- 添加避免时段
- 选择 AI 结果回填后是否自动导入审核排程

排程规则示例：

- 周会尽量固定在上午
- 高优先级会议避免安排在周五下午
- 同一高管连续会议之间至少保留 15 分钟缓冲

避免时段示例：

- `12:00 - 13:30`，原因：午休
- `18:00 - 20:00`，原因：高管私人时间

说明：

- 在 Electron 桌面版中可以保存 API Key 并直接提交后台 AI 任务
- 如果当前只是纯前端环境，仍可使用“复制完整 AI Prompt”的手工兜底方式

### 5.3 提交 AI 排程

完成配置后，可点击 `开始 AI 排程`。

系统会显示任务状态，例如：

- 排队中
- 提交中
- 等待结果
- 模型推理中
- 已完成
- 失败

AI 返回结果后，可直接导入系统，进入下一步人工审核。

### 5.4 手工兜底模式

如果暂时不走桌面端后台任务，也可以：

- 复制匿名输入 JSON
- 复制完整 AI Prompt
- 在外部 AI 工具中完成排程
- 将返回的 JSON 粘贴回系统
- 导入 AI 排程结果

## 6. 审核调整使用说明

`审核调整` 是人工确认排程的核心区域。

支持三种查看方式：

- 周视图
- 月视图
- 列表视图

主要能力：

- 查看 AI 提醒和冲突说明
- 查看本地检测出的时间冲突
- 拖拽移动会议时间
- 修改会议日期与时段
- 新增临时会议
- 将已有会议按某一日期临时插入审核区
- 删除审核区中的会议实例
- 标记 `预留`
- 标记 `锁定`
- 批量全部预留 / 全部取消预留
- 批量全部锁定 / 全部取消锁定
- 导入排程方案
- 导出排程方案

两个关键标记含义如下：

- `预留`：表示该会议已进入预留通知流程
- `锁定`：表示该会议时间已基本确认，不希望后续继续改动

建议做法：

1. 先处理 AI 提醒和冲突项
2. 再统一锁定已确认的会议
3. 最后进入最终检查

---

## 7. 最终检查使用说明

`最终检查` 用于核对“理论上应该出现的会议”与“当前审核区实际存在的会议”是否一致。

系统会按会议逐条检查：

- 预期发生次数
- 实际排入次数
- 是否存在遗漏
- 是否出现多排
- 是否存在需要人工核验的关联约束

常见状态包括：

- `待检查`
- `已确认`
- `疑似遗漏`
- `疑似误排`
- `次数异常`
- `需核验关联约束`

如果发现某条周期会议缺少某个应有日期，可直接在这里执行 `补回缺失实例`。

这一页建议作为排程完成前的最后一道人工关口。

---

## 8. 预留通知使用说明

`预留通知` 用于按会议自动生成通知文案，方便复制给高管、秘书或相关参会人。

主要功能：

- 按会议分组生成通知
- 自动汇总同一会议在当前阶段的多个安排
- 根据会议类型自动匹配模板
- 搜索通知内容
- 筛选 `全部 / 未发送 / 已发送`
- 一键复制通知文案
- 标记已发送

系统内置模板会自动替换以下变量：

- `【会议名称】`
- `【日期】`
- `【时间】`
- `【月份】`
- `【星期】`
- `【参会人】`
- `【高管名称】`
- `【秘书名称】`
- `【安排列表】`
- `【本阶段安排次数】`
- `【首次安排日期】`
- `【最后安排日期】`
- `【会议类型】`
- `【最近一次发生日期】`
- `【备注摘要】`

### 8.1 模板管理

系统支持两类模板：

- `内置模板`：只读，可复制、可隐藏
- `自定义模板`：可新增、编辑、删除

适用规则：

- 某个会议可绑定指定模板
- 如果未指定，系统会按会议名称和频率自动匹配默认模板

建议把常用的“1 对 1”“月度复盘”“通用预留”分别配置成独立模板，后续使用会更高效。

## 9. 记录模块使用说明

`记录` 页面用于查看操作审计日志，分为两类：

- `会议列表`
- `会议排程`

可记录的动作通常包括：

- 新建会议
- 编辑会议
- 删除 / 恢复 / 彻底删除
- 导入数据
- 调整排序
- 导入审核排程
- AI 排程结果回填
- 审核区调整操作

支持：

- 单条删除日志
- 清空全部日志

如果需要复盘某次排程是如何形成的，这个页面会很有帮助。

## 10. 数据保存与备份

### 10.1 自动保存

系统大部分业务数据默认自动保存在本地，包括：

- 会议主数据
- 排程数据
- 审核区状态
- 通知模板
- 日志记录
- AI 排程偏好

因此，日常使用中一般不需要额外点击“保存”。

### 10.2 导出系统备份

点击顶部 `系统备份 -> 导出系统备份`，可导出一个 JSON 文件，内容通常包含：

- meetings
- scheduled
- noticeTemplates
- disabledNoticeTemplateKeys
- logs
- aiState
- reviewState

建议：

- 每轮正式排程完成后导出一次
- 在批量改动会议资料前先导出一次

### 10.3 恢复系统备份

点击顶部 `系统备份 -> 恢复系统备份` 后，可选择 JSON 备份文件导入。

系统支持两种方式：

- `合并导入`：保留原有数据，只补充不存在的会议
- `覆盖导入`：用备份内容替换当前数据

建议优先使用 `合并导入`，只有在确认需要整体回滚时才使用 `覆盖导入`。

## 11. AI Key 与安全说明

在桌面版环境下，系统支持保存 `OpenAI` 或 `Gemini` 的 API Key。

实现方式：

- Key 保存在 Electron 用户数据目录中的本地配置文件
- 若系统支持安全加密，则优先使用 Electron `safeStorage`
- 如果运行环境不支持安全加密，可能以明文形式保存在本地配置中

因此建议：

- 仅在受控电脑上使用桌面版
- 避免把包含 Key 的系统目录同步到公共空间
- 离职交接或设备移交前及时清理本地 Key

## 12. 推荐操作规范

为了让排程更稳定，建议采用下面的工作习惯：

1. 先把会议库维护完整，再启动排程
2. 周期会议必须填写正确的频率和锚点日期
3. 导入历史记录后，再让 AI 排程，效果通常更好
4. AI 结果不要直接当最终版，必须经过“审核调整”和“最终检查”
5. 发送通知前，先统一确认“预留”标记
6. 每一轮正式发布前做一次系统备份导出

## 13. 常见问题

### 13.1 为什么没有生成某个周期会议？

可能原因：

- 会议被设为 `不定期`
- 周期规则填写错误
- 锚点日期不合理
- 当前排程时间范围未覆盖该会议的发生日期

### 13.2 为什么系统提示当前清单是旧版本？

说明你在生成待排程清单后，又修改了会议库中的资料。此时建议重新生成清单，避免 AI 基于旧数据工作。

### 13.3 为什么桌面版可以直接调 AI，而网页环境不行？

因为 API Key 保存、本地任务轮询和后台提交能力依赖 Electron 提供的桌面端接口。

### 13.4 为什么通知内容和预期不一样？

请检查：

- 会议是否绑定了正确模板
- 模板中使用了哪些变量
- 会议的参会人、备注、高管名称、秘书名称是否填写完整
